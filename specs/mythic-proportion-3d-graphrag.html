Purpose

The current mythic-proportion app is a working, independently-verified (102 tests green) local "second brain": a Karpathy-style LLM-Wiki over Markdown + [[wikilinks]], with a SQLite hybrid-search sidecar (sqlite-vec + FTS5), a six-verb CLI, and a FastAPI + vanilla-JS web UI whose graph tab is a flat 2D wikilink map. The "LLM Wiki Second Brain with 3D Interactive Knowledge Graph" blueprint describes a far more capable target: a GraphRAG memory engine (entities, relationships, claims, communities, community reports; global/local/DRIFT retrieval) rendered through a professional 3D WebGL knowledge graph, with an agent layer and an MCP interface so the brain is a memory substrate for external coding agents.

This plan closes that gap as a full greenfield rebuild to a course-corrected target (Vite + React + React Three Fiber on the existing Python/FastAPI core; sqlite-vec, not Neo4j/Qdrant; custom orchestration + optional PydanticAI, not LangGraph), migrating all existing data across behind an explicit behavior-parity gate, and adding a local privacy layer the blueprint never contemplated. The intended outcome: a private, offline-capable, visually premium knowledge graph that both a human and an agent can read and write — delivered incrementally, verified live in Chrome at every phase, and published as two separate public repositories.

Problem

The built app and the blueprint diverge on almost every axis above the storage line, and the current design cannot simply grow into the target:

  • Knowledge model is too thin. The schema is only pages / pages_fts / page_vectors / meta. There are no entities, typed relationships, text units, claims, communities, or community reports — so there is no semantic graph to render richly, and no GraphRAG retrieval (global/local/DRIFT, spreading-activation) is possible.
  • The graph view is a flat 2D wikilink map. api_graph emits page-nodes and resolved-wikilink edges onto a hand-rolled 2D canvas — no 3D, no instancing/LOD, no community structure, no scale headroom.
  • The frontend cannot carry a design system. 839 lines of imperative vanilla JS + 431 lines of hand-written CSS have no token layer, no component library, no build tooling — a ground-up design system is not maintainable there.
  • No privacy story. Every note is sent verbatim to a cloud LLM (AuthHub→DeepSeek). For a personal second brain this is the wrong default; there is no PII redaction, no local-embedding guarantee, no fully-offline option.
  • No agent layer, no interop. There is a single compile pass and a single query pass — no Extractor/Refiner/Librarian roles, and no MCP server, so no external agent can use the brain as memory.
  • No version control at all. Neither mythic-proportion/ nor the enclosing FABLE-HARNESS is a git repository, so there is nowhere to branch, review, or publish work.

Solution

Rebuild to the course-corrected target architecture, preserving every current behavior behind a parity gate and migrating all data across. The target keeps the Python core's ethos (files-first, lazily-imported optional extras, every web route a thin wrapper over the same building blocks the CLI uses, prompted strict-JSON) while adding six new capability layers:

  • GraphRAG data layer — new SQLite tables (entities, relationships, text_units, text_unit_entities, claims, communities, community_reports) + sibling vec0 vector tables, populated by prompted delimited-tuple extraction with an llm_cache for idempotent incremental indexing.
  • Retrieval engine — graspologic hierarchical Leiden communities + LLM community reports; global (map-reduce), local (seed→recursive-CTE expand), DRIFT (primer→local loop), and query-aware spreading-activation traversal, all over sqlite-vec + FTS5.
  • 3D frontend — Vite + React + r3f-forcegraph layout driving a custom InstancedMesh2 render layer (points-as-nodes, LOD, BVH frustum-cull, GPU picking) targeting ~50k nodes at 60fps, with a worker-frozen force layout, community hulls, hover/focus interactions, filters, a 2D fallback, and an accessible shadow-DOM.
  • Design system — three-tier OKLCH CSS-variable tokens (incl. a --graph-* family read straight into THREE.Color), shadcn/ui components, a Cmd+K command palette spine, motion tokens, light/dark.
  • Local privacy layer — Presidio + the OpenAI Privacy Filter for redact→(cloud or local)→rehydrate; bge-small-en-v1.5 local embeddings as default; an optional fully-offline Ollama/Qwen2.5 provider selected by a per-vault local: true flag.
  • Agent + interop layers — Extractor / Refiner / Librarian agents on a lean custom orchestrator (optional PydanticAI [agents] extra), and a FastMCP stdio server exposing read tools by default with opt-in transactional writes and a JSONL audit log.

All of it built by FABLE-HARNESS typed producers (engineer, designer, design-system-curator, security-reviewer), confirmed by verifier, and validated live in Chrome each phase — never Hydra, pair-programmer, smith, or eights.

Phase 5 — 3D graph frontend (~50k target) []

Worker-frozen layout, batched line edges with degree-based culling, SDF labels only for hovered/selected/high-degree nodes, community hulls, hover-neighbor highlight+fade, eased camera focus, filters, progressive disclosure, a 2D fallback mode, and an accessible parallel shadow-DOM.

Governance / build discipline. Every phase is built by FABLE-HARNESS typed producers only — engineer writes code, designer/design-system-curator own UX, data-modeler the schema, security-reviewer/ai-controls-author the trust surfaces, api-designer the contracts, docs-author the docs — with verifier independently confirming from disk and browser-validator/visual-regression-runner driving live Chrome. Reflexion ×1 per failing stage; escalate after 3 loops (N3). No Hydra, pair-programmer, smith, or eights (session constraint + project memory). Fable-5 (this plan) is planning only (N2) — never the build.

Course-corrections vs. the blueprint (carried from brief §7): Next.js → Vite+R3F served by FastAPI; 3d-force-graph → r3f-forcegraph + own InstancedMesh2; Qdrant/FAISS → sqlite-vec; Neo4j dropped; LangChain/LangGraph → custom + optional PydanticAI; MCP kept as a read-first product feature. New vs. blueprint: the entire privacy layer (P6).

Research provenance. Each capability rests on a cited research thread in specs/ROADMAP-BRIEF.md §6: privacy/embeddings/local-LLM (§6.1), GraphRAG-on-SQLite incl. schema/Leiden/query-modes/incremental (§6.2), 3D rendering at scale (§6.3), LangGraph+MCP (§6.4), ComfyUI automation (§6.5), design system (§6.6). Confidence note: the OpenAI Privacy Filter announcement page 403'd during research — its release-date/benchmark claims are corroborated via GitHub/HF + secondary coverage; validate the exact model card at P6.

Sequencing rationale. Design system (P1) precedes feature UI so nothing hard-codes style. Parity (P2) precedes new capability so the greenfield never regresses. Data layer (P3) precedes retrieval (P4) precedes the 3D view (P5) because each consumes the prior. Privacy (P6) can land any time after P2 but sits before the agent layer (P7) so agents inherit redaction. MCP (P8) is last of the core layers because it exposes everything beneath it. ComfyUI (P9) is optional and parallelizable. Cutover (P10) is the single retirement gate.
