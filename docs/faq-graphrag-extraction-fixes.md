# FAQ — GraphRAG Extraction Fixes (v0.7.0)

## General Questions

### Q: Do I need to do anything after updating to v0.7.0?

**A:** If your vault has never run extraction (`mythic index-graph`), no action is needed — just install the update.

If your vault **has** run extraction before v0.7.0, you should re-run extraction after updating to get:
1. Better-quality data (Defect #2 fix: now extracts from raw/ instead of lossy wiki/ pages)
2. Cleaner entities (Defect #3 fix: ~89% corruption removed)
3. No PII in stored data (Defect #4 fix: no unredacted PII in final records) — this benefit applies specifically if you have `settings.redaction_enabled=true`, but Defects #2 and #3 affect all vaults regardless of redaction setting

See **User Guide § Vault Data Cleanup** for step-by-step procedure.

### Q: Is this a breaking change?

**A:** No. All six CLI verbs (`init`, `ingest`, `query`, `lint`, `watch`, `serve`) and all existing API routes remain unchanged. The only new addition is the `POST /api/index-graph` endpoint and the new UI buttons/settings.

### Q: What does "REDACTED_PERSON_1" mean in my entity titles?

**A:** If you're seeing entity titles like "REDACTED_PERSON_1" or "REDACTED_EMAIL_2", your vault has corrupted data from before this fix. This is a placeholder token that was minted during extraction with `redaction_enabled=true`, but because redaction was scoped per-call (not per-turn), the placeholder was never rehydrated back to the real value in the final stored records.

After updating to v0.7.0, re-run extraction (see Runbook § Vault Data Cleanup); the new code will never create or persist these tokens.

### Q: I turned off redaction — am I safe?

**A:** Turning off redaction (`settings.redaction_enabled=false`) means:
- PII is NOT redacted before LLM calls
- The leak described in Defect #4 does NOT occur (because redaction is not running at all)
- BUT your vault content is sent unredacted to cloud LLM providers

This is a trade-off: if you have sensitive data, keep `redaction_enabled=true` and update to v0.7.0. If you don't have sensitive data, either setting is fine.

## Defect #1 Questions (Extraction Unreachable)

### Q: Where is the "Build Knowledge Graph" button?

**A:** In the **Ingest** tab of the web UI. It's a new button added in v0.7.0. If you don't see it, make sure:
1. You're running the latest code (v0.7.0+)
2. You're on the Ingest tab (not Search or Ask)
3. Browser console has no errors (F12)

### Q: How do I trigger extraction without the UI?

**A:** Use the CLI:
```bash
mythic index-graph --vault ./my-vault
```

Or poll the new API endpoint:
```bash
curl -X POST http://localhost:8765/api/index-graph \
  -H 'Content-Type: application/json' \
  -d '{"max_gleanings": 1}'
```

### Q: Can I set up auto-extraction after every ingest?

**A:** Yes. In the **Settings** tab, toggle **"Auto-build knowledge graph after ingest"** to ON. From then on, every time you upload files via the Ingest tab, extraction runs automatically.

Note: This incurs LLM cost (typically 100-500 calls per vault). It's OFF by default to avoid surprise charges.

## Defect #2 Questions (Lossy Source)

### Q: Why does my entity count suddenly jump after updating?

**A:** Before v0.7.0, extraction read the human-readable compressed `wiki/` pages (12,000 chars each) instead of the original `raw/` documents. After re-running extraction with the fixed code, you'll see more entities/relationships/claims because the extraction now operates on the full, uncompressed source material.

A 3,000-line document that previously yielded 42 text units might yield 200+ units after this fix.

### Q: Does this affect the wiki pages I browse?

**A:** No. The `wiki/` pages remain unchanged. This fix only changes what extraction reads as its input. The human-readable wiki pages are still generated by the compile pipeline and used for browsing in the Wiki tab.

### Q: My extracted entities look more detailed now — is that normal?

**A:** Yes, that's expected. Before this fix, entity descriptions were truncated because they were extracted from the already-compressed wiki pages. Now they're extracted from the full raw documents, so descriptions are longer and more complete.

## Defect #3 Questions (Corrupted Entities)

### Q: What was wrong with my entity descriptions?

**A:** Before v0.7.0, a prompt inconsistency caused entity and relationship descriptions to be corrupted. The prompt documentation said records should be delimited by `##`, but the few-shot example only showed `\n`-separated records. The LLM learned from the example and output `\n`-separated records; the parser only split on `##`, so multi-record completions became one blob, and each description absorbed the syntax of the next record.

Example: an entity description might end with `...key modeling.)\n("entity` instead of just `...key modeling.`

### Q: How do I know if my current data is corrupted?

**A:** Check the database:
```bash
sqlite3 ./my-vault/.index/index.db
> SELECT COUNT(*) FROM entities WHERE description LIKE '%)\n(%';
> .exit
```

If the count is > 0, you have corrupted data. Re-run extraction after updating to v0.7.0 (see Runbook).

### Q: My entities look fine — should I re-extract anyway?

**A:** If you're getting real, substantive entity descriptions (no `)\n(` syntax patterns), you might not have been affected. But to be safe and get the full benefit of Defect #2 fix (reading from raw/ instead of wiki/), re-extraction is still recommended.

## Defect #4 Questions (PII Leak)

### Q: Could my PII have leaked to the LLM?

**A:** Only if:
1. Your vault has `settings.redaction_enabled=true` (the Phase 6 default)
2. AND you ran extraction (`mythic index-graph`) before v0.7.0
3. AND extraction encountered a parse failure (causing a repair round) OR you ran with `--max-gleanings > 0` (causing gleaning rounds)

If all three are true, yes, real PII may have been sent to your LLM provider during the repair/gleaning rounds. After updating and re-running extraction with v0.7.0, this leak is closed.

### Q: What kind of PII could have leaked?

**A:** Any PII in your source documents that Presidio's analyzer detects:
- Names (people, organizations)
- Email addresses
- Phone numbers
- URLs
- Locations (cities, countries)
- Dates (if in personal context)

### Q: I have sensitive data — what should I do now?

**A:** 
1. Update to v0.7.0+
2. Review your vault's LLM provider logs/terms to understand what happened to the data that was sent
3. Re-run extraction with the fixed code (see Runbook § Vault Data Cleanup)
4. Verify the new data contains no PII tokens (see verification step in Runbook)

### Q: Does v0.7.0 completely prevent future leaks?

**A:** Yes, the redaction/rehydration architecture is now correct and tested. The new integration test (`tests/test_egress_gate.py`) exercises multi-round extraction with planted PII and verifies zero PII reaches the LLM at any point.

However, this only protects extraction (`mythic index-graph`). Compile and query operations have their own redaction paths; see Phase 6 spec for details.

### Q: Do I need the `[privacy]` extra installed?

**A:** For the leak to be closed, yes — `redaction_enabled=true` requires `[privacy]` to be installed. If redaction is enabled but `[privacy]` isn't installed, the code now raises an error instead of silently degrading.

To install:
```bash
pip install 'mythic-proportion[privacy]'
```

Or disable redaction:
```bash
# In .vault-meta/config.json:
# "redaction_enabled": false
```

## Testing & Verification

### Q: How do I verify my vault is clean after re-extraction?

**A:** Follow the Runbook § Vault Data Cleanup, Step 5-6:

1. **Database check** (no corruption/redacted tokens):
   ```bash
   sqlite3 ./my-vault/.index/index.db
   > SELECT COUNT(*) FROM entities WHERE title LIKE '%REDACTED_%';
   > SELECT COUNT(*) FROM entities WHERE description LIKE '%)\n(%';
   > .exit
   ```
   Both should return 0.

2. **Browser check** (real content):
   - Run `mythic serve --vault ./my-vault`
   - Go to Graph tab
   - Click on 2-3 entities
   - Verify real names and descriptions (not truncated, not corrupted syntax)

### Q: Are there new tests for these fixes?

**A:** Yes:
- **Defect #2**: No new test (it's architectural; tested by normal extraction flow)
- **Defect #3**: New test fixtures in `tests/test_graph.py` that use bare-`\n` record separation (like real LLM output)
- **Defect #4**: New integration test `tests/test_egress_gate.py` with planted PII, verifying zero PII in outbound payloads across all rounds

Run them:
```bash
pytest tests/test_graph.py::test_parse_bare_newline_records -v
pytest tests/test_egress_gate.py -v
```

### Q: Do I need to run tests before updating?

**A:** No, unless you're developing. End users just need to:
1. Update: `pip install --upgrade mythic-proportion`
2. Re-run extraction: `mythic index-graph --vault ./my-vault`
3. Verify (see Runbook)

## Deployment & Rollback

### Q: My organization uses Mythic Proportion internally — what's the migration plan?

**A:** 
1. **Stage 1 (immediate)**: Notify users of the security issue and the v0.7.0 fix
2. **Stage 2 (within 1 week)**: Provide runbook (included in this release) for vault re-extraction
3. **Stage 3 (within 2 weeks)**: Require upgrade to v0.7.0 for all extraction workflows
4. **Stage 4 (ongoing)**: Monitor logs for any PII-related incidents

### Q: What if I need to rollback?

**A:** See Runbook § Rollback. Downgrading to v0.6.x will restore the old (unfixed) code, but corrupted data that was already written won't be fixed. If you must rollback, do so before extracting with v0.7.0; once v0.7.0 extraction completes, the clean data is persisted and rolling back won't help.

### Q: How do I pin my version if I don't want to upgrade yet?

**A:** In your `requirements.txt` or `setup.py`:
```
mythic-proportion==0.6.x
```

However, this leaves the security issue open. We recommend updating as soon as possible.

## Glossary

- **Raw sources** (`raw/`): Original ingested documents, byte-for-byte as uploaded
- **Wiki pages** (`wiki/`): Compiled, human-readable, linked markdown pages (~12K chars each)
- **Extraction**: Running the GraphRAG entity/relationship/claim pipeline to build the knowledge graph
- **Repair round**: A single re-attempt when the LLM's first extraction parse failed
- **Gleaning round**: A recall-loop "did you miss anything?" round to find entities the first pass missed
- **Redaction**: Masking PII before sending data to the cloud LLM; masked values are replaced back (rehydrated) once the LLM responds
- **Rehydration**: Replacing masked PII tokens back with real values
- **REDACTED_* tokens**: Placeholder values minted during redaction (e.g., `[REDACTED_PERSON_1]`); should never persist in final data
